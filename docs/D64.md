# D64 Disk Image Format Specification

## Overview

The D64 format is a sector-by-sector dump of a Commodore 64 1541 floppy disk. It
is used by emulators to store and load disk images. This tool supports reading
D64 files to extract PRG files.

## File Structure

A D64 image contains the raw sectors of a 1541 disk. The disk is organized into
tracks and sectors:

| Track Range | Sectors per Track |
| ----------- | ----------------- |
| 1-17        | 21                |
| 18-24       | 19                |
| 25-30       | 18                |
| 31-35       | 17                |

Total: 683 sectors × 256 bytes = 174,848 bytes (standard D64)

Each sector is exactly 256 bytes.

## BAM (Block Availability Map)

Track 18, Sector 0 contains the BAM and disk header:

| Offset  | Size | Description                                     |
| ------- | ---- | ----------------------------------------------- |
| 0       | 1    | Next header track (0 = end of chain)            |
| 1       | 1    | Next header sector                              |
| 2       | 1    | DOS version type ($41 = DOS 2.0, $42 = DOS 2.1) |
| 3       | 1    | Double-sided flag                               |
| 4-143   | 140  | BAM entries for each track                      |
| 144-159 | 16   | Disk name (PETSCII, $A0 padded)                 |
| 160-161 | 2    | DOS version marker ($41 $00)                    |
| 162     | 1    | INODE byte                                      |
| 163-191 | 29   | Unused (typically $00)                          |
| 192-255 | 64   | Free sector count per track (2 bytes each)      |

## Directory Entries

Track 18, Sector 1 starts the directory. Directory entries are linked via
track/sector pointers:

| Offset | Size | Description                          |
| ------ | ---- | ------------------------------------ |
| 0      | 1    | Next entry track (0 = end)           |
| 1      | 1    | Next entry sector                    |
| 2      | 1    | File type                            |
|        |      | Bits 0-3: type (see below)           |
|        |      | Bit 6: locked flag                   |
|        |      | Bit 7: closed flag                   |
| 3      | 1    | File start track                     |
| 4      | 1    | File start sector                    |
| 5-20   | 16   | Filename (PETSCII, $A0 padded)       |
| 21-29  | 9    | Track/sector chain for file data     |
| 30-31  | 2    | File size in sectors (little-endian) |

### File Type Codes

| Code | Type | Description  |
| ---- | ---- | ------------ |
| 0    | DEL  | Deleted file |
| 1    | SEQ  | Sequential   |
| 2    | PRG  | Program      |
| 3    | USR  | User         |
| 4    | REL  | Relative     |

## File Data Storage

Files are stored as a chain of sectors. Each sector has:

| Offset | Size | Description          |
| ------ | ---- | -------------------- |
| 0      | 1    | Next track (0 = end) |
| 1      | 1    | Next sector          |
| 2-255  | 254  | File data            |

The last sector of a file has track = 0, and byte 1 indicates the number of
valid data bytes in that sector.

## PRG File Extraction

When extracting a PRG from D64:

1. The first 2 bytes of the extracted data are the load address (little-endian)
2. The remaining bytes are the program data
3. The file size in the directory is in sectors, not bytes

## Reading Notes

- Only PRG files are extracted by this tool (other types are skipped)
- Filenames use PETSCII encoding with $A0 padding
- DOS type $41 or $42 is required for valid D64
- Track 0 in directory chain indicates end of directory
